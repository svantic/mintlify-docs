---
title: "Svantic Sessions: Multi-Agent Conversation Context"
description: "Understand how Svantic sessions group agents and messages into a single conversation context, persist history, and enable real-time multi-agent orchestration."
---

A session is the unit of conversation in Svantic. It ties together a user, one or more agents, and a running history of messages and responses. Every `/ask` request you send belongs to a session, and the Mesh uses the session to determine which agents are available, maintain conversation context, and route capability invocations to the right agent instances.

Sessions are scoped to a tenant. All session history, traces, and events are isolated to the tenant that owns the session.

## Session lifecycle

<Steps>
  <Step title="Init">
    Create a session by sending a `POST /sessions/init` request with a `session_id`, `tenant_id`, and optional `user_id`. The Mesh validates agent admission, syncs the agent registry, and auto-binds the platform's internal agents. If you pass `self_agent_type` and `self_instance_id`, the requesting agent is also bound to the session immediately.

    ```bash
    POST /sessions/init
    {
      "session_id": "sess-acme-001",
      "tenant_id": "acme",
      "user_id": "user-42"
    }
    ```

    The response includes the list of agents currently bound to the session.
  </Step>
  <Step title="Ask">
    Send natural-language requests to `POST /sessions/{session_id}/ask`. The Mesh orchestrates one or more capability invocations across bound agents, streams events back in real time, and records the full exchange in session history.
  </Step>
  <Step title="Close or interrupt">
    When the conversation is complete, send `POST /sessions/close` to unbind all agents and persist the final state. If a response is taking too long, send `POST /sessions/interrupt` with the session ID to stop the in-flight stream immediately.
  </Step>
</Steps>

<Note>
  Session IDs are supplied by your application, not generated by Svantic. Use a value that is unique within your tenant — a UUID, a database record ID, or any string that is meaningful to your system.
</Note>

## How agents join a session

When a session is initialized, the Mesh automatically binds the platform's orchestration agents. Your own agents can join in two ways:

- **Self-join at init** — pass `self_agent_type` and `self_instance_id` in the init request to bind the calling agent immediately.
- **Admission-based binding** — the Mesh checks the tenant's allow-list and binds any admitted agent types that are currently registered.

<Warning>
  If you pass `self_agent_type` in the init request and that agent type is not on the tenant's allow-list, the session init will be rejected with a `403` error. See [Organizations and Tenants](/concepts/tenants) to manage the allow-list.
</Warning>

## Session history

Every message exchanged during a session — user inputs, agent responses, capability invocations, and results — is stored durably by the platform. This gives you a replayable record of the full conversation.

You can retrieve session history through the Gateway API. History is always scoped to a tenant, so queries only return sessions that belong to your organization.

## Real-time event streaming

The Mesh emits events as a session progresses. You can subscribe to these via the Dashboard or the streaming API. Key event types include:

| Event type | When it fires |
|---|---|
| `session.created` | A new session was initialized |
| `agent.bound` | An agent joined the session |
| `session.interrupted` | A user or API call interrupted an in-flight response |

Events carry `session_id` and `tenant_id` so you can correlate them with the correct session in your system.

## Multi-agent orchestration

A single `/ask` request can involve multiple agents. The Mesh presents all bound agents' capabilities to the LLM as a unified set of tools. The LLM decides which capabilities to call, in what order, and the Mesh dispatches those calls in parallel where possible.

Each capability invocation is a separate A2A call to the relevant agent instance. The results flow back to the LLM, which synthesizes a final response. The entire chain — from the user's message to the final answer — is captured as a single distributed trace that you can inspect in the Dashboard waterfall view.

<Tip>
  For long-running tasks, use the interrupt endpoint rather than letting the client time out. Interrupting a session cleanly stops the in-flight stream and records the interruption event, so the session history remains consistent.
</Tip>

<AccordionGroup>
  <Accordion title="Can I reuse a session across multiple conversations?">
    Yes. A session remains open until you explicitly close it. The Mesh keeps the agent bindings active and accumulates history. Whether you use one session per conversation or one long-lived session is a product decision — just be aware that history grows over time and is always available for replay.
  </Accordion>
  <Accordion title="What happens if the Mesh restarts mid-session?">
    Active session bindings are held in memory on the Mesh. If the Mesh restarts, in-flight bindings are lost and running streams are terminated. Session history persisted before the restart is preserved and remains queryable. You can re-initialize the session to restore agent bindings and continue.
  </Accordion>
  <Accordion title="Is session_id globally unique or only within a tenant?">
    Session IDs must be unique within a tenant. Two different tenants can each have a session with the same ID without conflict, because all session data is isolated by `tenant_id`.
  </Accordion>
</AccordionGroup>
